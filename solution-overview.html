For Product · PO & BA

One place to decide “who can do what — and when”

KAM is an access-control manager. It lets the business describe, in plain rules, exactly what each person is allowed to do with each kind of data — and it answers every “can this user do this?” question consistently, with a readable explanation for every yes and no.

🔐 Roles & permissions 🧩 Smart conditions 🧾 Explainable decisions 🏢 Per department & per record

1What KAM is (in one minute)

Skip the jargon — here is the whole idea.

Every product has the same recurring question: “Is this person allowed to do this action on this thing right now?” Today that logic is often scattered across code, spreadsheets and tribal knowledge. KAM makes it a single, visible set of rules that anyone can read.

You describe access using roles (job functions), permissions (allowed actions), and conditions (extra “only if…” checks). KAM then answers each request with a clear ALLOW or DENY — and shows its reasoning.

The problem it removes

  • Inconsistency — the same rule implemented three different ways in three features.
  • Over-granting — “just give them admin” because fine-grained access is too hard.
  • No audit story — nobody can explain why someone had access.
  • Slow change — a policy tweak means a code change and a release.

2The core building blocks

Five concepts cover everything. Once these click, the whole product makes sense.

👤

Users

The people (or services) making requests. Each user carries attributes like department, region and status.

🎭

Roles

Job functions such as Sales Rep or Finance Analyst. Roles inherit: a Sales Manager automatically gets everything a Sales Rep has, plus more.

📦

Resources

The things being protected. A resource type is a class (e.g. Invoices); a resource item is one concrete record (INV-1001) that has an owner.

✅

Permissions

An action a role may take on a resource type — read, update, approve, export… Set per cell in a simple matrix.

🧩

Conditions (ABAC)

“Only if…” rules layered on a permission — only if you own the record, only if it’s your department, only if you didn’t create it.

🧾

Decision & trace

Every check returns allow/deny plus the step-by-step reason, so support, audit and engineering all see the same story.

💡 Why “type vs. item” matters

Permissions are granted on a type (“Sales Reps can update Deals”), but conditions are checked against the item (“…only the Deals they own”). That split is what lets one rule safely cover thousands of records.

3How a single decision is made

Every request walks the same short funnel.

1
Who is asking?
KAM resolves the user’s roles — including everything inherited from parent roles.
2
Is there a matching permission?
It looks for a rule covering this action on this resource type. No rule → DENY (default-deny).
3
Is it an explicit block?
If any rule says DENY, it wins immediately — even over an allow elsewhere.
4
Do the conditions pass?
If the permission carries “only if…” conditions, each is checked against this record. All must pass.
✓
Final answer + reason
ALLOW or DENY, with the full trace of how it got there.

4The three golden rules

If a stakeholder remembers nothing else, remember these. They never change.

Rule 1

Default deny

If no rule grants access, the answer is no. Access is something you explicitly give, never assume.

Rule 2

Deny beats allow

An explicit block always wins. You can carve out a hard “never” that no other grant can override.

Rule 3

Closer role wins

A role’s own rule overrides what it inherits, so a specialised role can refine its parent without side effects.

5Worked example: the invoice approval

A real “separation of duties” rule, told as a story.

BO
Bob Ortiz · Financial Controller
Inherits: Finance Analyst → Employee. Department: Finance.

Request A — Bob tries to approve invoice INV-1002, which he created. His role does grant invoice approval, but the permission carries a “separation of duties” condition: you may not approve a record you own. Bob owns this one → DENY. KAM explains exactly why.

Request B — Bob approves invoice INV-1001, created by a colleague. Same permission, same condition — this time he is not the owner, so the condition passes → ALLOW.

✅ The point

One rule (“approve, only if not the owner”) enforces a finance-grade control across every invoice, with no per-record configuration and a clear audit reason for each outcome. The same pattern covers “only your department’s payroll”, “only records you own”, and more.

KAM authorization on Ory Keto — a hybrid ReBAC + ABAC design

Proposed by the technical lead for architecture review. We adopt Ory Keto (a Google Zanzibar implementation) as the relationship/permission engine, keep a relational database as the source of truth, and evaluate attribute conditions (ABAC) in a thin policy layer inside each service. Below: the model mapping, data flow, and the concrete Go integration pattern that protects every API.

Engine: Ory Keto Model: ReBAC + ABAC SoT: PostgreSQL Services: Go (net/http & gRPC) Invariants preserved

1Context, goals & non-goals

Goals

  • Centralised, fine-grained authorization shared by all Go services.
  • Preserve the prototype’s exact semantics: default-deny, deny-over-allow, nearer-role-over-ancestor.
  • Sub-10ms check latency at the edge of each request handler.
  • Policy changes without redeploys; full decision auditability.
  • Support both RBAC (role inheritance) and ABAC conditions (owner, dept, clearance, SoD, status, region).

Non-goals (for v1)

  • Replacing authentication — KAM assumes an authenticated subject id.
  • A general rules DSL — conditions are a fixed, typed catalogue (matching the prototype).
  • UI changes — the existing admin UI remains the policy authoring surface.
  • Cross-region active/active Keto replication (phase 2).

2Why Ory Keto — and the deliberate hybrid

Keto is excellent at relationships and inheritance, but it is not an attribute engine. We split responsibilities accordingly.

What Keto gives us

  • Zanzibar relation-tuples — object#relation@subject — a proven model for “who relates to what”.
  • Subject-sets express role membership and role inheritance natively (a role’s members include a parent role’s members).
  • Fast, horizontally scalable check and expand APIs (REST + gRPC).
  • Namespaces + the Ory Permission Language (OPL) to compose permissions from relations.

The ABAC gap → our split

Keto answers “is subject related to object?” It does not compare arbitrary attributes (subject.id == resource.owner, clearance ≥ sensitivity). The prototype’s conditions are exactly that.

Decision: Keto owns relationships & role hierarchy; a small condition evaluator in the service (reading attributes from the DB / request context) owns ABAC. The two compose as relationship AND conditions.

💡 Could we model ABAC as tuples instead?

Ownership and dept membership can be tuples (and we do that for the cheap, stable ones). But value comparisons like clearance ≥ sensitivity and separation-of-duties (subject ≠ owner) are awkward and churny as tuples. The hybrid keeps Keto’s data clean and keeps volatile attribute logic in code where it is testable.

⚠ Invariant guard

The three precedence rules are not left to Keto alone. The middleware orchestrates them explicitly: deny check → allow (Keto) check → condition (ABAC) check, with deny short-circuiting and a deny default. This is the contract that must hold and be unit-tested.

5Keto namespace configuration (OPL)

Roles, inheritance and the allow side of permissions, expressed in the Ory Permission Language.

namespaces.tsOPL
import { Namespace, SubjectSet, Context } from "@ory/keto-namespace-types"

// A role has members and may inherit a parent role's members.
class Role implements Namespace {
  related: {
    parents: Role[]
    members: (User | SubjectSet<Role, "members">)[]
  }
  permits = {
    // "is this subject a member, directly or via a parent role?"
    member: (ctx: Context): boolean =>
      this.related.members.includes(ctx.subject) ||
      this.related.parents.traverse((p) => p.permits.member(ctx)),
  }
}

class User implements Namespace {}

// One namespace per resource type (Invoice, Deal, Employee, Secret...).
class Invoice implements Namespace {
  related: {
    owner: User[]
    // each action is granted to a role's member set
    readers:   SubjectSet<Role, "members">[]
    approvers: SubjectSet<Role, "members">[]
    editors:   SubjectSet<Role, "members">[]
  }
  permits = {
    read:    (ctx: Context) => this.related.readers.includes(ctx.subject),
    approve: (ctx: Context) => this.related.approvers.includes(ctx.subject),
    update:  (ctx: Context) => this.related.editors.includes(ctx.subject),
  }
}

6Request flow — preserving the decision contract

The middleware reproduces the prototype’s decide() precedence step by step.

Caller
Handler request → middleware with subject, action, namespace, objectID
① Deny check
Middleware asks Keto for a deny relation. If present → DENY now (deny-wins, short-circuit).
② Allow check
Keto check(namespace, objectID, action, subject). Resolves role membership and inheritance server-side. Not allowed → DENY (default-deny).
③ Condition eval
If the grant carries ABAC conditions, the engine loads attributes from Postgres and evaluates each (owner, dept, clearance, sod, status, region). All must pass.
④ Verdict
ALLOW only if ② passed and every ③ condition passed; else DENY. Decision + trace emitted to the audit log.
✅ Mapping back to the prototype

Step ① = effect === "deny"; Step ② = effectiveUserGrants + role chain (now Keto); Step ③ = evalCondition AND-loop. Same three invariants, same trace shape.

7.1 — The authorizer (Keto client + condition engine)

authz/authorizer.goGo
package authz

import (
    "context"
    "fmt"

    rts "github.com/ory/keto/proto/ory/keto/relation_tuples/v1alpha2"
)

// Request is everything needed to make one decision.
type Request struct {
    SubjectID string // authenticated user id, e.g. "u2"
    Namespace string // resource type / Keto namespace, e.g. "Invoice"
    ObjectID  string // resource item id, e.g. "INV-1002"
    Action    string // "read" | "approve" | "update" ...
}

// Decision mirrors the prototype's decide() return.
type Decision struct {
    Allow  bool
    Reason string
    Trace  []string
}

type Authorizer struct {
    keto  rts.CheckServiceClient // Keto read API (gRPC)
    attrs AttributeStore       // reads subject/resource attrs from Postgres
    conds ConditionCatalog     // fixed catalogue: own, dept, sod, clr, active, region
}

func New(keto rts.CheckServiceClient, attrs AttributeStore, conds ConditionCatalog) *Authorizer {
    return &Authorizer{keto: keto, attrs: attrs, conds: conds}
}

// keto reports whether one relation holds (role membership + inheritance
// are resolved server-side by Keto's subject-set expansion).
func (a *Authorizer) ketoCheck(ctx context.Context, ns, obj, rel, subj string) (bool, error) {
    res, err := a.keto.Check(ctx, &rts.CheckRequest{
        Tuple: &rts.RelationTuple{
            Namespace: ns,
            Object:    obj,
            Relation:  rel,
            Subject:   rts.NewSubjectID("user:" + subj),
        },
    })
    if err != nil {
        return false, fmt.Errorf("keto check: %w", err)
    }
    return res.Allowed, nil
}

// Authorize enforces the three invariants in order:
//   1) explicit deny wins      2) allow via Keto (else default-deny)
//   3) every ABAC condition must pass
func (a *Authorizer) Authorize(ctx context.Context, r Request) (Decision, error) {
    d := Decision{Reason: "default deny"} // zero value = deny

    // (1) deny-wins: a deny relation short-circuits everything.
    if denied, err := a.ketoCheck(ctx, r.Namespace, r.ObjectID, "deny:"+r.Action, r.SubjectID); err != nil {
        return d, err
    } else if denied {
        d.Reason = "explicit deny takes precedence"
        d.Trace = append(d.Trace, "deny rule matched")
        return d, nil
    }

    // (2) allow via Keto (role membership + inheritance). No allow ⇒ default-deny.
    rel := actionRelation(r.Action) // "approve" -> "approvers"
    allowed, err := a.ketoCheck(ctx, r.Namespace, r.ObjectID, rel, r.SubjectID)
    if err != nil {
        return d, err
    }
    if !allowed {
        d.Reason = "no matching grant (default deny)"
        return d, nil
    }
    d.Trace = append(d.Trace, "grant matched via Keto")

    // (3) ABAC conditions attached to this grant — all must pass (AND).
    bag, err := a.attrs.Load(ctx, r.SubjectID, r.Namespace, r.ObjectID)
    if err != nil {
        return d, err
    }
    for _, c := range a.conds.For(r.Namespace, r.Action) {
        if ok, why := c.Eval(bag); !ok {
            d.Reason = "condition failed: " + why
            d.Trace = append(d.Trace, d.Reason)
            return d, nil // still deny
        }
        d.Trace = append(d.Trace, "condition passed: "+c.Name())
    }

    d.Allow, d.Reason = true, "all checks passed"
    return d, nil
}

7.2 — HTTP middleware that protects handlers

authz/middleware.goGo
package authz

import "net/http"

// Protect wraps a handler, requiring `action` on the resource identified
// by `namespace` + a path/route value. Subject comes from auth context.
func (a *Authorizer) Protect(namespace, action string, objectFrom func(*http.Request) string) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, req *http.Request) {
            subject, ok := SubjectFromContext(req.Context()) // set by your auth layer
            if !ok {
                http.Error(w, "unauthenticated", http.StatusUnauthorized)
                return
            }

            dec, err := a.Authorize(req.Context(), Request{
                SubjectID: subject,
                Namespace: namespace,
                ObjectID:  objectFrom(req),
                Action:    action,
            })
            if err != nil {
                http.Error(w, "authorization unavailable", http.StatusServiceUnavailable) // fail closed
                return
            }
            if !dec.Allow {
                audit(subject, namespace, action, dec) // emit trace for audit
                http.Error(w, "forbidden: "+dec.Reason, http.StatusForbidden)
                return
            }
            next.ServeHTTP(w, req) // only reached on ALLOW
        })
    }
}

7.3 — Wiring it into your routes

main.goGo
func main() {
    conn, _ := grpc.Dial("keto:4466", grpc.WithInsecure())
    az := authz.New(rts.NewCheckServiceClient(conn), pg.NewAttributeStore(db), authz.DefaultCatalog())

    invoiceID := func(r *http.Request) string { return chi.URLParam(r, "id") }

    r := chi.NewRouter()
    // GET  /invoices/{id}          -> needs read on Invoice
    r.With(az.Protect("Invoice", "read", invoiceID)).Get("/invoices/{id}", getInvoice)
    // POST /invoices/{id}/approve -> needs approve (+ separation-of-duties cond)
    r.With(az.Protect("Invoice", "approve", invoiceID)).Post("/invoices/{id}/approve", approveInvoice)

    http.ListenAndServe(":8080", r)
}
ℹ Operational notes

Fail closed on Keto/DB errors (503, never allow). Cache hot check results per request and short-TTL across requests. For gRPC services, the same Authorize call goes in a UnaryInterceptor instead of HTTP middleware. Batch checks with Keto’s BatchCheck where a handler touches many objects.

8Source of truth & tuple sync

Postgres = source of truth

The relational DB stays authoritative for policy authoring (roles, grants, conditions) and attributes (user dept/clearance/status, item owner/dept/sensitivity). The admin UI writes here.

Keto holds a derived projection: the relationship tuples needed for fast checks. It is a cache/index, not the master.

Keeping Keto in sync

  • On policy/ownership change, emit a domain event (outbox pattern).
  • A projector translates the change into Keto write/delete tuple ops (idempotent).
  • A periodic reconciler diffs DB-derived tuples vs. Keto and repairs drift.
  • Attributes are not projected to Keto — they are read live by the condition engine.
